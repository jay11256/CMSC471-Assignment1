Introduction to the Dataset

We decided to use the Daily Weather in the U.S., 2017 dataset.
This dataset contains daily weather data from various weather stations in 2017. Exploratory data analysis was used to gain a general understanding of the entire dataset (code can be found in "prelim.ipynb" on the github). We determined that the scope of the entire dataset was too large as it contained daily data from hundreds of weather stations from different states, territories, and provinces, so we decided to focus on Maryland weather stations.
The dataset contained the data for 6 MD weather stations. After plotting these locations on a map, we determined that 2 pairs of locations were in very close proximity, so we eliminated the lesser known location of each pair, resulting in the 4 locations used above.

The original dataset included variables such as snowfall, snow depth, and average temperature. We decided to exclude these from the visualization variables/dropdowns for several reason. First, the dataset had no data for these variables for a significant amount of days. Snowfall and snow depth were understandable as this region isn't known for snowfall, but it was strange that average temperature wasn't recorded. We could've imputed data based on the max and min temperatures of each day, but we decided against it as we wanted to visualize the given dataset without modifying data. Second, the dataset doesn't have data for the full year. As noted at the top of this webpage, the dataset only contains data ranging from January to September, missing multiple winter months which may have seen snowfall. We decided to exclude snow data as a whole as there was only data for half of the winter season.

Rationale for Design Decisions

Our web visualization approach was designed to effectively display the daily weather data across multiple stations in Maryland. We chose to use scatter plots to represent the relationship between different weather variables. Each axis can be customized by the user, allowing for a flexible exploration of the data, and easy on-demand plots of various relationships among variables in the dataset. The size of the points in the scatter plot is also adjustable, providing an additional dimension for analysis. The color coding by season allows for easy identification of patterns in the data. We use the plus icon across all plots to allow for easy understanding of our brushing/linking feature, which is activated by click and drag. The box can then be moved across the graph to make quick inspection easy.

Some patterns we want users to be able to identify easily are how certain weather trends are associated with certain seasons. Being able to highlight specific seasons and brush certain clusters of data points makes it easy to observe these patterns. For instance, users can set the X and Y axis to be wind measurements, and by selecting different seasons to highlight, they can see that winter and spring tend to have higher wind speeds in all locations. Being able to brush also allows users to identify outliers in data. When setting the axes to temperature measurements, users can see that the Patuxent graph has several outliers where there appears to be significant temperature deviations during the summer. By brushing over these points, users can see that these extremities aren't reflected in the other locations, suggesting that the Patuxent station may have had faulty measurements or truly extreme weather.

Development Process

Our development process was iterative, involving multiple stages of design, implementation, and testing. As a team, we started by exploring the dataset and identifying key variables for visualization. Jason Liu was responsible for the initial data exploration and variable selection, filtering the data for our visualization. We then designed the user interface to allow for flexible exploration of the data, including customizable axes and adjustable point sizes. Raghav Chakravarthy implemented the brushing and linking feature and was responsible for the structure of the graph UI in a grid-like format, which was implemented to enable users to interact with the data dynamically. Throughout the development process, we conducted usability tests and made adjustments based on feedback. Charles Phan was responsible for the initial template of the UI and the coloring of the data points by date, using a color gradient. Overall, the development process was collaborative and user-focused, taking 15 man hours of development to complete. Most of the time was spent fixing the UI structure to make it visually understandable and appealing.
